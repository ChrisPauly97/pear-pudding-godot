## Context

Shared dungeon crawls (TID-380) generate byte-identical dungeons for all co-op peers using a seeded generator, but today a player who loses a PvE battle is ejected to the single-player defeat screen, leaving the party without them and turning a shared challenge into a punishment. By adding downed/rescue mechanics, a loss inside a dungeon becomes a dramatic moment: the fallen player becomes a spirit at the defeat spot, froze in place or allowed limited movement, while teammates race to revive them via an interact prompt (E key + tap for parity). This creates real party stakes and tactical decision-making — do we push forward or backtrack to revive? A solo fallback ensures no one is permanently trapped: if all party members fall downed or a timer expires, the downed player respawns at the dungeon entrance, allowing the run to continue or gracefully dissolve. The mechanic is surgical: it only activates inside dungeon maps (`map_name.starts_with("dungeon_")`) during active co-op (`_coop_active`), leaving single-player and non-dungeon co-op entirely unchanged.

**Related systems:** Battle loss routing today triggers the single-player defeat path (SceneManager routing, player-home respawn from docs/agent/player-home.md); multi-map co-op transitions (TID-355); avatar sync payload (GID-096, AvatarSync carries map name in 5th element); pure wire helpers in `game_logic/net/` following unit-tested conventions.

## Research Notes

**Defeat routing interception:** Currently, `BattleManager` or `GameState` signals a battle loss, which routes through `SceneManager` to either a game-over screen or player-home bed respawn (see docs/agent/player-home.md for the current flow). The interception point is in WorldScene's battle loss signal handler. Add a guard: `if _coop_active and current_map.starts_with("dungeon_"):` send the player to the downed state instead of the normal defeat path. Single-player and non-dungeon co-op (e.g., shared wilderness exploration) always use the standard defeat path.

**Downed state & avatar:** When downed, create a tinted/desaturated RemotePlayer-like avatar at the defeat spot (or reuse the player's existing sprite with a shader tint). Decide in the Plan phase: is the downed player frozen in place or can they move freely (flavorful but adds complexity)? Interactions (chests, NPCs, doors) are blocked — only movement and the revive interact are available. The downed player sees a UI indicator (e.g., "Waiting for rescue…") and a timer counting down to auto-respawn.

**Downed/revived wire protocol:** Define a minimal pure wire helper in `game_logic/net/` following the existing pattern (AvatarSync, unit-tested). The payload carries: `{peer_id: int, downed: bool, position: Vector3}` or extend the AvatarSync 5th-element map field to include a `"downed"` flag. On receive, non-authority peers update the avatar state and UI accordingly. Authority (host) arbitrates revive validity to avoid races — if a revive request arrives after the downed player has already respawned or timed out, silently discard it.

**Revive interaction:** A teammate interacts with the downed player (E key or tap the avatar with an interact prompt overlay, same pattern as chest/NPC prompts for parity). This triggers a local interact event that the interacting peer broadcasts via RPC to the host. The host validates: is the revived peer actually downed? Are they within interaction range? If valid, flip the downed flag and apply any reposition logic (e.g., move to the reviver's side). Emit a brief success toast and play a "revival" sound for both players.

**Solo fallback:** Maintain a `_downed_timers: Dictionary[int, float]` tracking time-to-auto-respawn for each downed peer. If the timer expires, auto-respawn at the dungeon entrance (same spawning logic as entering the dungeon normally). Additionally, if all connected peers are downed simultaneously, trigger the fallback respawn to avoid a soft-lock. The respawn teleports the player(s) back to the dungeon's entrance and clears the downed state.

**Dungeon entrance detection:** Dungeons are generated by `DungeonGen.generate(map_name, seed)` (TID-380, docs/agent/multiplayer-coop.md). They have a defined spawn location (passed to `WorldScene._spawn_player` on entry). Store this as `_dungeon_spawn_pos: Vector3` when entering a dungeon; on auto-respawn, teleport there.

**Authority:** Only the host updates downed state in `SessionState`/`SessionStore` (mirroring the AvatarSync mirror pattern from the multiplayer docs). Clients send revive requests as RPCs; the host validates and broadcasts the state change. This prevents race conditions where two clients try to revive the same player simultaneously.

**SceneManager routing:** Check how `SceneManager` currently routes battle loss and game-over today (see docs/agent/player-home.md). The interception happens *before* that routing, so the normal flow is never triggered inside a dungeon co-op session.

**Headless import & preload rules:** New wire helpers and any new scene references must follow CLAUDE.md preload rules; all `.gd` files must compile under `godot --headless --editor --quit`.
